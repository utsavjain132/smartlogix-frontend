## Summary

This phase replaces legacy `.css` styling with a modern utility-first architecture based on Tailwind CSS v4 and `shadcn/ui`. The application needs `jsconfig.json` configured for path aliases to support shadcn components in a non-TypeScript Vite environment. The legacy PostCSS configuration should be removed to rely fully on `@tailwindcss/vite`. Since all custom CSS files are being deleted, elements must be rewritten using either raw Tailwind utilities or shadcn base components (`Button`, `Input`) to prevent visual regressions while standardizing on a high-trust SaaS theme using deep blues and the Inter typeface.

**Primary recommendation:** Establish the Tailwind v4 base config via `src/index.css` using inline `@theme` directives, bootstrap shadcn via `init -d`, install fundamental components (`button`, `input`), and iteratively map legacy `.css` rules to equivalent Tailwind classes directly in the `.jsx` files.

## Architecture Patterns

### Recommended Project Structure
```
src/
├── assets/          # Static assets
├── components/      # React components
│   ├── ui/          # Generated by shadcn/ui
│   └── ...          # App components
├── lib/             # Utilities
│   └── utils.js     # Generated by shadcn/ui (cn function)
├── pages/           # Route views
└── index.css        # Global CSS + Tailwind config
```

### Pattern 1: Tailwind v4 Configuration via CSS
**What:** Tailwind v4 moves configuration from `tailwind.config.js` entirely into CSS using the `@theme` directive.
**When to use:** Global theming setup for colors and fonts.
**Example:**
```css
/* Source: Tailwind CSS v4 documentation */
@import "tailwindcss";

@theme inline {
  --font-sans: "Inter", sans-serif;
  --color-primary: oklch(0.45 0.15 260); /* Deep Blue */
  --color-primary-foreground: oklch(0.98 0.01 240); /* Slate 50 */
}
```

### Pattern 2: Component Migration Strategy
**What:** Replacing legacy CSS classes with Tailwind equivalents inline to ensure no visual regressions.
**When to use:** When converting `.jsx` files previously relying on `.css` imports.
**Example:**
```jsx
// Legacy
// <button className="btn-primary">Get Started</button>

// Migrated to Shadcn
import { Button } from "@/components/ui/button";
<Button>Get Started</Button>

// Or Migrated to Tailwind Utilities
// <Link to="/signup" className="inline-flex items-center justify-center rounded-md text-sm font-medium bg-primary text-primary-foreground hover:bg-primary/90 h-10 px-4 py-2">Get Started</Link>
```

### Anti-Patterns to Avoid
- **Keeping `postcss.config.js` or `tailwind.config.js`:** Tailwind v4 manages its own compilation through the Vite plugin. Mixing PostCSS configs causes conflicts.
- **Using relative paths for Shadcn:** Shadcn relies heavily on the `@/` path alias. Failing to configure `jsconfig.json` and `vite.config.js` for alias resolution breaks component generation.
- **Partial CSS Deletion:** Leaving some custom CSS while migrating others causes unpredictable styling cascades. Migrate fully per component and delete the original `.css`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Button States | Custom `button` CSS with hover/focus variants | `shadcn/ui` Button | Shadcn handles all interactive states, accessibility, and variant mapping (`ghost`, `outline`, `default`) natively. |
| Input Styling | Custom form inputs with focus rings | `shadcn/ui` Input | Accessible forms with consistent focus rings and disabled states across browsers are deceptively difficult to build perfectly. |
| Class Merging | Template literals ``className={`btn ${isActive ? 'active' : ''}`}`` | `cn()` utility (`clsx` + `tailwind-merge`) | Resolves Tailwind specificity conflicts seamlessly (provided by `shadcn init` in `lib/utils.js`). |

**Key insight:** The foundation of the new design system requires zero custom CSS. Standardizing completely on utility classes and Shadcn components ensures future phases (Data Tables, Modals) can snap into place visually without manual styling.

## Common Pitfalls

### Pitfall 1: Shadcn Init Without Path Aliases
**What goes wrong:** Running `npx shadcn@latest init` fails with "No import alias found".
**Why it happens:** The project is standard JavaScript (no TypeScript) and lacks a `jsconfig.json`.
**How to avoid:** Create a `jsconfig.json` defining `@/*` -> `./src/*` and configure `resolve.alias` in `vite.config.js` *before* running shadcn.
**Warning signs:** Shadcn CLI throwing error about aliases during preflight.

### Pitfall 2: `import.meta.dirname` Support in Vite Config
**What goes wrong:** Using `__dirname` in `vite.config.js` throws an error because the project is "type": "module".
**Why it happens:** `__dirname` is a CommonJS variable and undefined in ES modules.
**How to avoid:** Use `path.resolve(import.meta.dirname, "./src")` or `new URL("./src", import.meta.url).pathname`. Node v20.11+ supports `import.meta.dirname`.

### Pitfall 3: Broken Global Scope
**What goes wrong:** Removing `App.css` or `index.css` entirely removes essential HTML/body resets.
**Why it happens:** Blindly deleting all CSS files instead of analyzing their contents.
**How to avoid:** Migrate `body` backgrounds, fonts, and core animations into the `@layer base` of `index.css` instead of deleting them outright.

## Code Examples

### Vite Path Alias Configuration
```javascript
// vite.config.js
import path from "path";
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  plugins: [react(), tailwindcss()],
  resolve: {
    alias: {
      "@": path.resolve(import.meta.dirname, "./src"),
    },
  },
});
```

### Essential JSConfig
```json
// jsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```
